01 — 문제 정의

매 턴, 처음부터
다시 계산한다

코딩 에이전트는 파일 읽기 → 코드 수정 → 테스트 실행을 수십 번 반복합니다. 같은 코드 파일이 매 턴 context에 등장하지만, Serving Layer는 이를 매번 처음부터 prefill합니다.

"코딩 에이전트의 멀티턴 세션에서 중복 코드 블록이 반복 등장함에도 Serving Layer가 매 턴 전체를 prefill하여 불필요한 지연 비용이 발생한다."

vLLM prefix caching은 hit rate 75.8%임에도 Vanilla decode 대비 TTFT 차이가 단 0.8 ms에 불과합니다. 코드 블록 구간에서는 캐시가 사실상 작동하지 않기 때문입니다.

02 — 방법론

세 가지 핵심 아이디어

기존 prefix caching이 반복 코드를 "다른 것"으로 인식할 때, SynTree KV는 재사용 가능한 구조적 단위로 인식합니다.

🌳

AST 서브트리 단위 KV 재사용

Tree-sitter로 실시간 파싱하여 함수·클래스·import 단위를 추출합니다. 깊이 ≥ 2, 토큰 수 ≥ 8인 노드를 색인화하여 재등장 시 즉시 캐시에서 호출합니다.

🔑

α-rename 정규화 + BLAKE3 지문

변수명을 VAR_0, VAR_1…으로 치환하여 구조만 남기고 BLAKE3으로 32byte 지문을 생성합니다. 이름이 달라도 구조가 같으면 같은 블록으로 인식합니다.

📐

RoPE Re-positioning

위치가 달라진 KV를 그대로 삽입하면 attention이 왜곡됩니다. 위치 오프셋 Δ를 이용한 회전 행렬 R(Δ)로 보정하여 정확한 재사용을 보장합니다.

α-rename 정규화 예시
# Turn 1에서 등장한 코드
def calc(price, tax):
    total = price * tax
    return total

# Turn 4에서 등장한 코드 — 이름은 다르지만 구조는 동일
def compute(amount, rate):
    result = amount * rate
    return result

# α-rename 적용 → 두 함수 모두 동일한 정규화 결과
def VAR_0(VAR_1, VAR_2):
    VAR_3 = VAR_1 * VAR_2
    return VAR_3

# BLAKE3 해시 → a3f9c2d1... == a3f9c2d1...  →  캐시 히트 ✓

RoPE Re-positioning 보정 수식

재사용 KV는 원래 위치에서 계산된 값입니다. 새 context에서 다른 위치에 삽입될 때 위치 오프셋 Δ를 이용해 Key 벡터를 회전 보정합니다. PromptCache(미처리), CacheBlend(부분)와 달리 완전 보정을 수행합니다.

Knew = R(Δ) · Kcached
Δ = 새 위치 − 원래 위치
R(Δ) = RoPE 회전 행렬

분석 결과 — 4가지 조건 모두 GO ✓

06 — 시스템 구조

전체 파이프라인

토큰화부터 LRU eviction까지 단일 흐름으로 처리됩니다.

1

토큰화 + Tree-sitter 증분 파싱

입력 코드를 실시간 파싱하여 AST를 생성합니다. {FunctionDef, ClassDef, ImportStmt, CallExpr} 노드를 깊이 ≥ 2, 토큰 수 ≥ 8 조건으로 선별합니다.

2

α-rename → BLAKE3 지문 → SessionCacheTable 조회

서브트리별 fingerprint를 산출하고 캐시 테이블에서 히트 여부를 판단합니다.

3a

적중 (hit) — prefill skip + RoPE-shifted KV splice

prefill 재계산을 생략하고 저장된 KV에 R(Δ) 위치 보정을 적용하여 삽입합니다.

3b

미적중 (miss) — 정상 prefill 후 SessionCacheTable 등록

새로 계산된 KV를 지문을 키로 캐시에 저장합니다.

4

Decode + 발열 관리

splice된 슬롯을 일반 슬롯과 동일하게 attention에 노출합니다. 32토큰마다 발열 지표를 polling합니다.

5

턴 종료 — LRU eviction

SessionCacheTable을 512 MB 이내로 유지합니다. 가장 오래된 항목부터 제거합니다.

한계점 및 개선 방향

#1 RoPE 정확도

보정이 완벽하지 않으면 pass@1이 하락할 수 있습니다. YaRN·PoSE 계열 ablation 실험과 Δ가 작은 구간만 허용하는 보수적 정책으로 보완합니다.

#2 vLLM 블록 정렬

AST 서브트리와 vLLM 블록 단위(16~32토큰) 경계 불일치 오버헤드가 발생할 수 있습니다. 패딩 토큰으로 블록 단위에 맞게 정렬하여 해소합니다.

#3 Python 한정

현재 Python에만 적용됩니다. tree-sitter-javascript, tree-sitter-java 등 언어별 문법 파일을 추가하여 다중 언어로 확장할 예정입니다.
